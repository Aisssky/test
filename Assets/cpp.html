🛡️ 防卫式声明（Include Guard）

它的作用是防止同一个头文件被多次 #include 导致重复定义。

C++ 的 #include 本质是文本粘贴，假设没有防卫：

  • A.h 包含 B.h
  • C.h 包含 B.h
  • Main.cpp 包含 A.h 和 C.h

编译 Main.cpp 时，B.h 的内容被粘贴了两次，导致类/结构体重复定义，编译报错。UHT（Unreal Header Tool）在处理 .generated.h 文件时，依赖头文件不被重复解析，没有防卫会导致 UHT 报错。

防卫式声明通过宏标记"我已经被包含过了"，第二次 #include 时整个文件内容被预处理器跳过。

经典写法示意

经典写法：

#ifndef MY_HEADER_H
            #define MY_HEADER_H

            // 头文件内容

            #endif

或者现代写法（UE5 模板生成的文件天然就有）：

#pragma once

            // 头文件内容

⚡ InLine

本意是告诉编译器：不要为这个函数创建独立的调用地址，把函数体直接"复制粘贴"到调用处。

inline int Add(int a, int b)
            {
            return a + b;
            }

            // 调用处
            int result = Add(3, 5);

            // 编译后实际变成
            int result = 3 + 5;  // 函数体被直接展开，没有函数调用开销

函数调用本身有开销：压栈、跳转、返回、出栈。对于极短小、频繁调用的函数，这个开销比函数本身还贵。inline 就是为了消除这个开销。

inline 现在更核心的意义已经不是"展开优化"，而是：

允许同一个函数定义出现在多个编译单元（.cpp）里，而不报重复定义错误

// MathUtils.h
            inline int Add(int a, int b)  // 没有 inline → 多个 .cpp include 此头文件时报错
            {
            return a + b;
            }

            加了 inline，链接器会自动合并重复定义，只保留一份。

🎮 在 UE5 中的常见形式

UE5 里你会见到两种主要写法：

  1. 直接写在头文件里的短函数 
    FORCEINLINE float GetHealth() const
            {
            return Health;
            }
  2. .inl 文件

    UE5 里有些复杂的模板实现会单独放在 .inl 文件里，再在头文件末尾 include 进来：

    // MyClass.h
            #pragma once

            class MyClass { ... };

            #include "MyClass.inl"  // inline 实现单独存放，保持头文件整洁

UE5 的 FORCEINLINE

UE5 不直接推荐写 inline，而是用自己的宏：

FORCEINLINE float GetSpeed() const { return Speed; }

区别在于：

行为
inline 建议编译器展开，编译器可以拒绝
FORCEINLINE 强制展开（底层是 __forceinline / __attribute__((always_inline))）

📝 一句话总结

inline 的现实意义有两个：①消除短函数的调用开销；②让函数定义可以安全地住在头文件里。在 UE5 中你更多会见到 FORCEINLINE，是它的加强版。

UE5 的"内联生成代码"机制

内联生成代码 · 编译单元合并示意

这其实是一个宏展开成路径字符串，再被 #include 引入的技巧：

// 宏展开后大概等价于
            #include "InventoryItemDefinition.gen.cpp"

.gen.cpp 是什么

平时熟悉的是 UHT 生成的 .generated.h，但其实 UHT 同时还会生成 .gen.cpp，里面包含：

  • UClass 的反射注册代码
  • UFUNCTION 的函数绑定
  • 序列化、属性元数据等样板代码

正常情况下这个 .gen.cpp 是作为独立编译单元参与构建的。

这是 UE5.1 引入的优化手段，叫做 Inlined Generated Code：

方式 行为
传统方式 .gen.cpp 单独编译，是一个独立的编译单元
Inline 方式 .gen.cpp 被 include 进你的 .cpp，合并成同一个编译单元

合并后的好处：

  1. 减少编译单元数量 → 构建系统要处理的文件变少
  2. 编译器可以跨两个文件做优化，因为它们现在是同一份翻译单元
  3. Unity Build 更友好，整体编译速度提升

规则和 .generated.h 必须最后一个类似——这行也要放在 .cpp 文件的 include 列表末尾。

它本质是把 UHT 生成的反射样板代码内联进你的 .cpp，而不是单独编译，目的是减少编译单元、提升构建速度。是 UE5 的编译优化手段，Lyra 示例项目里大量使用，属于"最佳实践"写法。
